ВВЕДЕНИЕ
     В последние годы мобильные приложения занимают всё более значимое место в жизни современного человека [8, 16]. Одной из наиболее востребованных категорий являются казуальные игры, которые позволяют пользователям отвлечься, расслабиться и провести время с интересом. В рамках данной курсовой работы была поставлена задача разработать мобильное приложение — игру «Баскетбол» для платформы Android [1, 3, 11].
     Актуальность выбранной темы обусловлена ростом популярности мобильных игр и необходимостью приобретения студентами практических навыков проектирования и реализации программных продуктов для мобильных устройств [2, 3, 16]. Мобильные игры, в том числе спортивные аркады, востребованы среди широкой аудитории пользователей благодаря простоте освоения и возможности быстрого включения в игровой процесс. Разработка собственной игры позволяет не только закрепить теоретические знания, полученные в ходе обучения, но и получить опыт решения реальных задач программирования, проектирования интерфейса и тестирования программного продукта.
     Целью курсовой работы является создание работоспособного мобильного приложения-игры, в которой реализованы базовые элементы игрового процесса: управление мячом, начисление очков, выбор игровых объектов, а также система достижений. Для достижения поставленной цели были определены следующие задачи:
1. Изучить предметную область и существующие аналоги;
2. Сформулировать техническое задание и требования к разрабатываемому приложению;
3. Спроектировать архитектуру и интерфейс приложения;
4. Реализовать основные компоненты игры;
5. Провести тестирование и анализ полученных результатов.
     Объектом исследования является процесс разработки мобильных игр для платформы Android. Предметом исследования — методы и средства реализации игровых механик и пользовательского интерфейса в мобильных приложениях.
     Методы исследования включают анализ предметной области, сравнительный анализ аналогов, проектирование архитектуры, реализацию на языке Java с использованием Android SDK, а также функциональное и пользовательское тестирование приложения на различных устройствах.
     Практическая значимость работы заключается в получении студентом опыта самостоятельной разработки мобильного приложения, освоении инструментов Android-разработки, а также в создании готового программного продукта, который может быть использован для дальнейшего развития или публикации в магазине приложений.
     Структура работы включает введение, четыре главы, заключение, список литературы и приложения. В первой главе рассматривается анализ предметной области, существующих решений и формулируется техническое задание. Во второй главе описывается процесс проектирования архитектуры и интерфейса приложения. Третья глава посвящена реализации основных компонентов игры. В четвертой главе приведены результаты тестирования и анализ работы приложения.


1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ
     1.1. Обзор рынка мобильных игр и актуальность темы
     Мобильные игры занимают значительную долю рынка программного обеспечения для смартфонов [16]. По данным аналитических агентств, спортивные аркады и казуальные игры стабильно входят в топ популярных приложений среди пользователей Android. Особое место занимают игры, связанные с баскетболом, благодаря простоте освоения и возможности соревноваться с самим собой или с другими игроками.
     1.2. Анализ существующих аналогов
     В ходе анализа рынка были рассмотрены следующие популярные мобильные игры баскетбольной тематики:
     - «Basketball Stars» — аркадная игра с возможностью соревноваться с другими игроками, отличается развитой системой достижений, разнообразием игровых режимов и кастомизацией персонажей.
     - «Dunk Hit» — простая казуальная игра, в которой основной задачей является забрасывание мяча в корзину с увеличением сложности по мере прохождения.
     - «Street Basketball Association» — спортивная аркада с элементами управления командой, наличием различных режимов и достижений.
     Общие черты аналогов:
     - Простое и интуитивное управление (обычно свайпы или нажатия);
     - Минималистичный, но яркий интерфейс;
     - Система поощрений и достижений;
     - Возможность выбора игровых объектов (мячи, персонажи);
     - Сохранение прогресса пользователя.
     В отличие от сложных многопользовательских игр, в рамках данной курсовой реализуется однопользовательская аркада с акцентом на простоту освоения, стабильность работы и базовые механики: управление мячом, подсчёт очков, выбор мяча, система достижений, музыкальное сопровождение, сброс прогресса.
     1.3. Техническое задание и требования к приложению
     Для систематизации требований к разрабатываемому приложению приведены таблицы функциональных и нефункциональных требований:
1. Управление мячом: Пользователь может совершать броски с помощью касаний экрана
2. Подсчёт очков: За каждое попадание начисляются очки, отображается текущий счёт
3. Выбор мяча: Пользователь может выбрать один из нескольких мячей, выбор сохраняется
4. Система достижений: Открытие достижений при выполнении условий (например, 50 очков)
5. Музыкальное сопровождение: Фоновая музыка с возможностью отключения
6. Сброс прогресса: Очистка достижений и выбранного мяча по запросу пользователя
7. Адаптация интерфейса: Корректная работа на устройствах с разными разрешениями экрана:
1. Минималистичный интерфейс: Простой, интуитивно понятный дизайн
2. Оффлайн-работа: Приложение не требует подключения к интернету
3. Быстрый отклик: Время реакции на действия пользователя ? 0,2 сек
4. Локальное хранение данных: Использование SharedPreferences для сохранения данных
5. Стабильность: Отсутствие сбоев и зависаний при длительном использовании
6. Энергоэффективность: Минимальное потребление ресурсов устройства
     Каждое требование сопровождается проверкой его реализации на этапе тестирования (см. главу 4).
     1.4. Выводы по главе
     В результате анализа предметной области и существующих решений были определены основные требования к разрабатываемому приложению. Принятые проектные решения направлены на создание простой, стабильной и удобной игры, соответствующей ожиданиям пользователей и возможностям студента 3 курса. В следующей главе будет подробно рассмотрен процесс проектирования архитектуры и интерфейса приложения.


2. ПРОЕКТИРОВАНИЕ ПРИЛОЖЕНИЯ
     2.1. Архитектура приложения и выбор технологий
     В качестве платформы для реализации игры выбран Android, что обусловлено широкой распространённостью устройств и доступностью инструментов разработки. В качестве языка программирования используется Java, а для построения пользовательского интерфейса — стандартные средства Android SDK. Архитектура приложения построена по принципу Model-View-Controller (MVC), что позволяет разделить логику управления, отображения и хранения данных. В качестве модели выступают классы, отвечающие за хранение состояния игры, пользовательских настроек и достижений. Представление реализовано с помощью пользовательских интерфейсов (Activity и View), а контроллеры обеспечивают обработку событий и взаимодействие между компонентами.
     Взаимодействие между основными компонентами представлено на схеме (см. Приложение А, рисунок 1). Каждый экран реализован отдельной Activity, что соответствует рекомендациям Android-разработки [1].
     2.1.2. Описание основных классов
     MainActivity — основной управляющий класс, реализующий главное меню. Содержит методы для обработки нажатий на кнопки, перехода к другим Activity, а также инициализации музыкального сопровождения. Взаимодействует с BallSelectActivity, GameActivity, AchievementsActivity и ResetProgress через Intent.
     BallSelectActivity — реализует экран выбора мяча. Содержит методы отображения списка мячей, обработки выбора пользователем, сохранения выбранного мяча в SharedPreferences. Взаимодействует с MainActivity для возврата к главному меню.
     GameActivity — основной игровой экран. Инициализирует GameView, обрабатывает жизненный цикл игры, подсчёт очков, завершение раунда. Взаимодействует с AchievementsActivity для отображения новых достижений.
     GameView — кастомный View для отрисовки игрового поля, мяча, корзины. Обрабатывает касания экрана, реализует физику движения мяча, проверку попадания в корзину. Взаимодействует с GameActivity для передачи информации о результатах броска.
     AchievementsActivity — экран достижений. Загружает и отображает список достижений, полученных пользователем. Использует данные из SharedPreferences.
     MusicPlayer — отдельный класс для управления воспроизведением фоновой музыки. Позволяет запускать, останавливать и переключать треки в зависимости от состояния приложения.
     ResetProgress — реализует логику сброса пользовательских данных. При подтверждении пользователем очищает SharedPreferences и уведомляет MainActivity о необходимости обновления интерфейса.
     2.1.3. Взаимодействие между компонентами
     Переходы между экранами реализованы с помощью Intent. Для передачи данных между Activity используются дополнительные параметры Intent или SharedPreferences. Например, при выборе мяча BallSelectActivity сохраняет идентификатор выбранного мяча, который затем используется в GameActivity для отображения соответствующего спрайта (см. листинг 1 в Приложении Б).
     2.2. Структура данных и хранение пользовательской информации
     2.2.1. Хранение пользовательских данных
     SharedPreferences используется для хранения:
     - выбранного мяча (ключ "selected_ball");
     - количества набранных очков (ключ "score");
     - списка полученных достижений (ключ "achievements");
     - настроек звука (ключ "music_enabled").
     Данные сохраняются при каждом изменении состояния, что обеспечивает их сохранность даже при аварийном завершении приложения.
     2.2.2. Структура данных
     Для хранения достижений используется структура данных в виде списка строковых идентификаторов. Для каждого достижения предусмотрено отдельное условие получения, реализованное в GameActivity (см. листинг 2 в Приложении Б).
     2.3. Сценарии использования приложения
     2.3.1. Подробные пользовательские сценарии
     Сценарий 1. Запуск приложения:
1. Пользователь запускает приложение.
2. Отображается главное меню (см. скриншот 1 в Приложении А).
3. Автоматически запускается фоновая музыка.
     Сценарий 2. Начало новой игры:
1. Пользователь нажимает кнопку "Начать игру".
2. Осуществляется переход на GameActivity.
3. Отображается игровое поле, мяч, корзина, текущий счёт (см. скриншот 3 в Приложении А).
4. Пользователь совершает броски, управляя мячом с помощью касаний.
5. При попадании мяча в корзину увеличивается счёт, при промахе — игра завершается.
6. После завершения игры отображается результат и предлагается начать заново или вернуться в меню.
     Сценарий 3. Выбор мяча:
1. Пользователь выбирает пункт "Выбрать мяч" в главном меню.
2. Осуществляется переход на BallSelectActivity.
3. Отображается список доступных мячей с изображениями (см. скриншот 2 в Приложении А).
4. Пользователь выбирает мяч, информация сохраняется в SharedPreferences.
5. Осуществляется возврат в главное меню.
     Сценарий 4. Просмотр достижений:
1. Пользователь выбирает пункт "Достижения" в главном меню.
2. Осуществляется переход на AchievementsActivity.
3. Отображается список полученных достижений с описаниями (см. скриншот 4 в Приложении А).
     Сценарий 5. Сброс прогресса:
1. Пользователь выбирает пункт "Сбросить прогресс".
2. Отображается диалог подтверждения.
3. При подтверждении все данные очищаются, пользователь уведомляется всплывающим сообщением.
     2.3.2. Дополнительные пользовательские сценарии
     Сценарий 6. Ошибка загрузки музыки:
1. Пользователь запускает приложение на устройстве с отсутствующим аудиофайлом.
2. Приложение пытается воспроизвести музыку.
3. При ошибке выводится уведомление, музыка отключается (см. листинг 9 в приложении А).
     Сценарий 7. Смена ориентации экрана:
1. Во время игры пользователь поворачивает устройство.
2. Приложение сохраняет состояние (счёт, положение мяча) через onSaveInstanceState.
3. После поворота состояние восстанавливается, игра продолжается с того же момента.
     Сценарий 8. Восстановление после сбоя:
1. В случае аварийного завершения приложения при следующем запуске предлагается восстановить последнюю сессию.
2. Если пользователь соглашается, данные загружаются из SharedPreferences.
     2.4. Описание интерфейса и навигации
     2.4.1. Интерфейс главного экрана
     Главный экран содержит четыре кнопки: "Начать игру", "Выбрать мяч", "Достижения", "Сбросить прогресс". В верхней части экрана отображается название игры. Кнопки выполнены в едином стиле, имеют крупный шрифт и хорошо различимы на фоне. В нижней части экрана отображается текущий выбранный мяч (иконка).
     2.4.2. Интерфейс экрана выбора мяча
     Экран содержит список мячей, каждый из которых представлен изображением и кратким описанием. При выборе мяча он выделяется цветной рамкой. В нижней части экрана расположена кнопка возврата в главное меню.
     2.4.3. Интерфейс игрового экрана
     Игровое поле занимает основную часть экрана. В верхней части отображается текущий счёт, в центре — корзина, в нижней части — мяч. Управление осуществляется с помощью касаний: пользователь тянет мяч вверх, задавая силу и направление броска. После броска отображается анимация полёта мяча и результат (попадание или промах).
     2.4.4. Интерфейс экрана достижений
     Список достижений представлен в виде вертикального списка. Для каждого достижения отображается иконка, название и краткое описание условия получения. Полученные достижения выделяются цветом.
     2.4.5. Цветовая схема, шрифты и доступность интерфейса
     Интерфейс приложения выполнен в светлой цветовой гамме с акцентами на кнопках и игровых объектах. Основной цвет — белый (#FFFFFF), кнопки и активные элементы выделяются синим (#2196F3) и оранжевым (#FF9800). Для текста используется шрифт Roboto, размер 18sp для заголовков и 14sp для основного текста. Все элементы имеют достаточный контраст для удобства восприятия. Кнопки крупные, с увеличенным отступом для удобства нажатия. Интерфейс адаптирован для слабовидящих: все элементы снабжены описаниями для screen reader, поддерживается масштабирование шрифтов.
     2.4.6. Адаптация интерфейса под разные устройства
     Для обеспечения корректного отображения на различных устройствах используются ConstraintLayout и отдельные ресурсы для разных разрешений (drawable-mdpi, hdpi, xhdpi). Интерфейс тестировался на смартфонах и планшетах с разными диагоналями. Все элементы масштабируются пропорционально размеру экрана, что обеспечивает удобство использования на любом устройстве (см. листинг 10 в приложении А).
     2.5. Обработка ошибок и устойчивость приложения
     2.5.1. Обработка ошибок
     В приложении реализована обработка следующих ошибок:
     - Ошибка чтения/записи SharedPreferences: при невозможности сохранить или загрузить данные пользователю выводится уведомление с предложением повторить действие. В случае повторных неудач предлагается сбросить настройки до значений по умолчанию.
     - Ошибка загрузки графических ресурсов: если не удаётся загрузить изображение, подставляется стандартная иконка.
     - Ошибка воспроизведения музыки: при отсутствии аудиофайла или ошибке MediaPlayer музыка отключается, пользователь информируется через диалоговое окно.
     - Некорректные действия пользователя: при попытке выбрать недоступный мяч или совершить действие вне игрового процесса выводится предупреждающее сообщение.
     - Сбои при смене ориентации экрана: реализована обработка событий onSaveInstanceState и onRestoreInstanceState, что позволяет сохранять состояние игры при повороте устройства.
     - Восстановление после аварийного завершения: при следующем запуске приложение проверяет целостность данных и предлагает восстановить последнюю сессию.
     2.5.2. Устойчивость к сбоям
     Все критические операции сопровождаются логированием ошибок для последующего анализа и улучшения стабильности приложения (см. листинг 9 в приложении А).

3. РЕАЛИЗАЦИЯ ПРИЛОЖЕНИЯ
     3.1. Общая структура проекта
     Проект реализован в виде набора Java-классов, каждый из которых отвечает за отдельный функциональный модуль или экран приложения. В корневой папке проекта размещены основные Activity, вспомогательные классы, ресурсы (графика, музыка) и файлы конфигурации. Структура проекта соответствует стандартам Android-разработки, что облегчает поддержку и масштабирование приложения.
     3.2. Реализация главного экрана (MainActivity)
     MainActivity отвечает за отображение главного меню и обработку пользовательских событий. В методе onCreate() происходит инициализация интерфейса, установка слушателей нажатий на кнопки, запуск фоновой музыки (см. листинг 1 в приложении Б). Для перехода к другим экранам используются Intent:
     Intent intent = new Intent(this, GameActivity.class);
     startActivity(intent);
     Также реализована обработка события onResume() для возобновления музыки и onPause() для её остановки. При нажатии на кнопку "Сбросить прогресс" вызывается диалог подтверждения, после чего происходит очистка данных через ResetProgress.
     3.3. Экран выбора мяча (BallSelectActivity)
     BallSelectActivity реализует отображение списка доступных мячей с помощью RecyclerView. Для каждого элемента списка отображается изображение и название мяча. При выборе мяча его идентификатор сохраняется в SharedPreferences (см. листинг 2). После выбора осуществляется возврат в главное меню. Интерфейс адаптирован под разные разрешения экранов с помощью ConstraintLayout.
     SharedPreferences prefs = getSharedPreferences("settings", MODE_PRIVATE);
     SharedPreferences.Editor editor = prefs.edit();
     editor.putInt("selected_ball", ballId);
     editor.apply();
     3.4. Игровой экран (GameActivity и GameView)
     GameActivity отвечает за инициализацию игрового процесса, создание экземпляра GameView, обработку событий завершения игры. В GameView реализована отрисовка игрового поля, мяча, корзины, анимация движения мяча, обработка касаний экрана (onTouchEvent). Физика броска реализована с учётом начальной скорости, угла и силы, рассчитываемых на основе жеста пользователя (см. листинг 3).
     @Override
     public boolean onTouchEvent(MotionEvent event) {
     if (event.getAction() == MotionEvent.ACTION_UP) {
     // Вычисление силы и направления броска
     // Запуск анимации полёта мяча
     }
     return true;
     }
     При попадании мяча в корзину увеличивается счёт, проверяется выполнение условий для получения достижений. После завершения игры отображается диалог с результатом и предложением начать заново или вернуться в меню (см. скриншот 3 в приложении А).
     3.5. Экран достижений (AchievementsActivity)
     AchievementsActivity загружает список достижений из SharedPreferences и отображает их в виде списка с иконками и описаниями. Полученные достижения выделяются цветом. Для каждого достижения реализована проверка условий получения в GameActivity (см. листинг 4).
     if (score >= 50 && !achievements.contains("score_50")) {
     achievements.add("score_50");
     // Сохранение и уведомление пользователя
     }
     3.6. Управление музыкой (MusicPlayer)
     Класс MusicPlayer реализует воспроизведение фоновой музыки с помощью MediaPlayer. Музыка запускается при входе в приложение и останавливается при выходе. Реализована возможность отключения музыки через настройки (см. листинг 5). В случае ошибки загрузки файла музыка отключается автоматически.
     MediaPlayer player = MediaPlayer.create(context, R.raw.music);
     player.setLooping(true);
     player.start();
     3.7. Сброс прогресса (ResetProgress)
     Класс ResetProgress реализует очистку всех пользовательских данных в SharedPreferences. После сброса пользователь уведомляется всплывающим сообщением (Toast), интерфейс обновляется (см. листинг 6).
     SharedPreferences.Editor editor = prefs.edit();
     editor.clear();
     editor.apply();
     3.8. Работа с ресурсами и адаптация интерфейса
     Графические ресурсы (иконки мячей, корзины, фон) размещены в папке res/drawable. Для разных разрешений используются отдельные папки (drawable-mdpi, hdpi, xhdpi). Интерфейс построен с помощью ConstraintLayout, что обеспечивает адаптацию под различные размеры экранов. Все текстовые строки вынесены в ресурсы (strings.xml) для удобства локализации.
     3.9. Хранение данных и обработка ошибок
     Для хранения пользовательских данных используется SharedPreferences. Все операции чтения и записи сопровождаются проверкой успешности выполнения. В случае ошибки отображается уведомление и предлагается повторить действие. Критические ошибки (например, отсутствие доступа к памяти) корректно обрабатываются, приложение завершает работу без потери данных.
