## 1. Введение

Мобильные игры занимают доминирующую долю среди скачиваемых приложений в мире: согласно статистике Google Play за 2024 год, на игры приходится более 70% выручки магазина. При этом жанр аркадных игр остаётся одним из наиболее востребованных благодаря простоте управления и высокой повторяемости — механика «ещё одного броска» обладает высоким удерживающим потенциалом.

Данная выпускная квалификационная работа посвящена разработке мобильного приложения **BasketMars** — аркадной баскетбольной игры для платформы Android. Приложение реализует физически корректный бросок мяча в кольцо, несколько игровых режимов с разными правилами, в том числе онлайн-матчмейкинг с реальным игроком через Firebase Firestore, прогрессивную систему сложности, систему достижений и инвентарь скинов, двуязычный интерфейс и режим дуэли против ИИ-бота.

Принципиальное отличие проекта от коммерческих аналогов — отказ от сторонних игровых движков (Unity, Unreal, libGDX). Весь рендеринг, физика, анимации и UI реализованы средствами нативного Android SDK (Java, Canvas API), что позволяет детально изучить архитектурные паттерны Android-разработки и низкоуровневые механизмы отрисовки.

Проект опубликован на GitHub (открытый исходный код) и собирается в виде APK под Android 5.0+ (API 21+). Приложение полностью функционирует в офлайн-режиме; онлайн-возможности (облачный лидерборд, PvP) активируются при наличии рабочего Firebase-проекта.

## 4. Описание разработанного приложения

### 4.1 Функциональность

**BasketMars** — аркадная игра, в которой игрок бросает баскетбольный мяч в кольцо жестом свайп. Приложение включает:

- **4 игровых режима:** Аркада (бесконечный с нарастающей сложностью), На время (60 сек с бонусными очками), Дуэль (против ИИ-бота), **Онлайн PvP** (реальный игрок через Firebase Firestore)
- **Система сложности в Аркаде:** движущееся кольцо (с ≥6 очков), первое горизонтальное препятствие (с ≥3), второе препятствие (с ≥10), ускорение всех элементов по кривой до +105%
- **Комбо-система:** счётчик серии попаданий, бонусное очко в TIMED при серии ≥3, визуальное усиление с цветовыми уровнями (жёлтый → оранжевый → фиолетовый), тряска экрана при серии ≥5
- **Инвентарь:** 3 варианта мяча, 3 варианта сетки, 3 фоновых градиента
- **8 достижений:** разблокируются каждые 3 очка, открывают скины
- **Лидерборд:** хранится локально (Room) с опциональной синхронизацией в облако (Firebase Firestore), 4 вкладки режимов
- **Настройки:** имя игрока, музыка, вибрация, язык (RU/EN), Google-аккаунт
- **Музыкальный плеер:** 2 фоновых MP3-трека с кнопками управления (prev/next)
- **Анимации:** конфетти при победе (ConfettiView), дрожание кольца при попадании, spring-физика сетки, вспышка при ударе
- **Матчмейкинг:** очередь ожидания соперника с 30-секундным таймаутом и автоматическим откатом на ИИ-бота
- **Сброс прогресса:** полная очистка SharedPreferences и Room через отдельный экран

### 4.2 Экраны приложения

## 5. Архитектура и техническое решение

### 5.1 Общая архитектура

Приложение спроектировано по **многослойной (Layered) архитектуре**, разделённой на четыре слоя:

```
┌──────────────────────────────────────────────────────────┐
│                       UI Layer                            │
│  Activity · BottomSheetDialog · RecyclerView              │
│  MatchmakingActivity · InventoryActivity · etc.           │
├──────────────────────────────────────────────────────────┤
│                  Game Engine Layer                         │
│  GameView — Canvas, Physics, Animation, HUD, Combo        │
│  ConfettiView — победная анимация                         │
├──────────────────────────────────────────────────────────┤
│               Network / AI Layer                           │
│  MatchClient (interface — паттерн Strategy)               │
│  ├── DemoDuelClient    (ИИ-бот, полностью локальный)      │
│  └── OnlinePvpClient   (Firestore real-time PvP)          │
├──────────────────────────────────────────────────────────┤
│                   Data Layer                               │
│  Room DB (SQLite)  ·  Firebase Firestore                  │
│  SharedPreferences ·  LeaderboardRepository               │
│  CloudLeaderboardRepository · AuthManager                 │
└──────────────────────────────────────────────────────────┘
```

Каждый слой взаимодействует только со смежным. `GameView` не знает ни о Room, ни о Firestore напрямую — доступ к данным проходит через Activity и Repository.

### 5.2 Паттерны проектирования

| Паттерн | Где применён |
|---|---|
| **Repository** | `LeaderboardRepository`, `CloudLeaderboardRepository` — скрывают источник данных (Room vs Firestore) от Activity |
| **Singleton** | `AuthManager.getInstance()`, `BasketballGameApp` (Application-класс), `MusicPlayer` |
| **Strategy** | `MatchClient` (интерфейс) — заменяет ИИ-бот (`DemoDuelClient`) на онлайн-клиент (`OnlinePvpClient`) без изменения `GameView` |
| **Observer** | `MatchClient.Listener` — асинхронные callbacks от сетевого слоя в `GameView` |
| **Factory Method** | `GameMode.fromName(String)` — безопасное создание enum из строки с дефолтом ARCADE |
| **Template Method** | `onDraw(Canvas)` в `GameView` — фиксированный порядок слоёв отрисовки: фон → препятствия → кольцо → сетка → мяч → HUD → эффекты |

### 5.3 Жизненный цикл игры

```
MainActivity.showModeDialog()
  ├── startGame(ARCADE / TIMED)  ──────────────► GameActivity.onCreate()
  │                                                   └─► GameView.init() → немедленный старт
  ├── startGame(ONLINE_DUEL)  ───────────────► GameActivity.onCreate()
  │                                                   └─► GameView.init() → DemoDuelClient.connect()
  │                                                            └─► onConnected() → запуск бота
  └── startActivity(MatchmakingActivity)
        └─► Firestore matchmaking_pool
              ├── [найден соперник] ───────────► GameActivity (ONLINE_PVP, roomId + myRole)
              │                                       └─► OnlinePvpClient.connect()
              │                                                └─► score1/score2 real-time sync
              └── [таймаут 30 сек] ────────────► GameActivity (ONLINE_DUEL fallback)
```

## 6. Реализация игровых режимов

### 6.1 Аркада (ARCADE)

Бесконечный режим с прогрессивным усложнением — основная и наиболее насыщенная часть игры.

**Прогрессия сложности** управляется методом `applyArcadeDifficulty()`, вызываемым после каждого гола. Сложность растёт по плавной кривой `t = score / 30f` (0..1), ограниченной максимумом:

| Порог очков | Событие |
|---|---|
| ≥ 3 | Появляется первое горизонтальное препятствие (движется вправо) |
| ≥ 6 | Кольцо начинает двигаться по горизонтали |
| ≥ 10 | Появляется второе препятствие (движется влево) |

Скорость каждого элемента масштабируется:
- Кольцо: базовая скорость × (1 + 1,05 · t)
- Препятствие 1: базовая × (1 + 1,20 · t)
- Препятствие 2: базовая × (1 + 1,25 · t)

Скорость каждого элемента масштабируется:
- Кольцо: базовая скорость × (1 + 1,05 · t)
- Препятствие 1: базовая × (1 + 1,20 · t)
- Препятствие 2: базовая × (1 + 1,25 · t)

При максимальном значении t=1 (≥30 очков) скорость кольца увеличивается в 2,05 раза, препятствий — в 2,2–2,25 раза. Препятствия позиционируются так, чтобы не перекрывать сам обод кольца (защитная зона вокруг кольца).

**Рисунок 1. Прогрессия сложности в режиме Аркада (относительная скорость, t = score/30):**

```
Скорость  │
   205% ──┤                                              ████ кольцо
   200% ──┤                                         ████████
   185% ──┤                                    █████████████ препятствие 1
   175% ──┤                                ████████████████
   165% ──┤                           █████████████████████ препятствие 2
   150% ──┤                      ██████████████████████████
   140% ──┤                 ███████████████████████████████
   125% ──┤            ████████████████████████████████████
   100% ──┼────────────────────────────────────────────────
          0     3     6    10    15    20    25   30+ очков
          │     │     │     │
          │     │     │     └── второе препятствие
          │     │     └──────── кольцо начинает двигаться
          │     └────────────── первое препятствие
          └──────────────────── старт (нет препятствий)
```

**Комбо-система:**
- `comboStreak` — счётчик непрерывных попаданий
- При серии ≥ 2 на экране появляется анимированный счётчик с цветовым уровнем:
  - 2–4 попадания: жёлтый (`#FFEE58`)
  - 5–7: оранжевый (`#FF9800`)
  - ≥ 8: фиолетовый (`#E040FB`)
- При серии ≥ 5: тряска экрана (`shakeUntilMs`) с нарастающей амплитудой
- Любой промах сбрасывает серию в ноль

### 6.2 На время (TIMED)

60-секундный режим с усиленным акцентом на мастерство:
- Обратный таймер отображается на HUD
- **Бонусные очки:** при серии ≥ 3 попаданий подряд каждое следующее попадание приносит **+2 очка** вместо +1 — механика поощряет стабильную игру
- Режим без препятствий и движущегося кольца — чистая проверка броска
- По окончании — результат сравнивается с личным рекордом в лидерборде; экран результата блокирует новые броски

### 6.3 Дуэль с ботом (ONLINE_DUEL)

60-секундный матч против ИИ-бота:
- Бот подключается через интерфейс `MatchClient` (реализация: `DemoDuelClient`)
- Рядом с кольцом отображается «призрак» бота — анимированный мяч в полупрозрачном цвете
- После истечения времени определяется результат: победа / поражение / ничья
- При победе — анимация конфетти (`ConfettiView`) и тактильный отклик
- Статистика дуэлей хранится в `PlayerStats` (`duelWins`, `duelLosses`, `duelBest`)

### 6.4 Онлайн PvP (ONLINE_PVP)

60-секундный матч против реального игрока через Firebase Firestore.

**Рисунок 2. Диаграмма последовательности онлайн-матчмейкинга:**

```

     │                         │                          │                      │
     │── нажать «Онлайн PvP» ─►│                          │                      │
     │                         │── PUT matchmaking_pool ─►│                      │
     │                         │   {userId, name, ts}     │                      │
     │                         │                          │◄─ PUT matchmaking ───│
     │                         │                          │   pool (Игрок B)     │
     │                         │◄── snapshot: появился ───│                      │
     │                         │    соперник              │                      │
     │                         │── CREATE matches/roomId ►│                      │
     │                         │   {p1Id,p2Id,status}     │──── snapshot ───────►│
     │                         │── DELETE из pool ────────►│◄─── DELETE из pool ─│
     │                         │                          │                      │
     │◄── GameActivity ────────│                          │         GameActivity ►│
     │    ONLINE_PVP           │                          │         ONLINE_PVP   │
     │                         │                          │                      │
     │── sendSnapshot(score) ──────────────────────────── ►│                      │
     │                         │                          │──── onGhostSnapshot ►│
     │◄─ onGhostSnapshot ──────────────────────────────── ─│◄─── sendSnapshot ───│
     │   (счёт соперника)      │                          │                      │
     │                         │                          │                      │
     │── status: "finished" ───────────────────────────── ►│◄── "finished" ───────│
     │   (конец игры)          │                          │                      │
```

**Матчмейкинг (`MatchmakingActivity`):**
1. Игрок помещается в очередь `matchmaking_pool/{userId}` в Firestore
2. Отображается spinner «Поиск соперника…»
3. Firestore-слушатель (`addSnapshotListener`) ожидает появления соперника в коллекции
4. При нахождении соперника создаётся документ матча `matches/{roomId}` со статусом `waiting`
5. Автотаймаут 30 сек: если соперник не найден — запускается режим «Дуэль с ботом» без потери игрового опыта

**Игровой процесс (`OnlinePvpClient`):**
- Каждый игрок знает свой `roomId` и `myRole` (1 или 2)
- Счёт записывается в поля `score1`/`score2` документа матча
- Частота записи ограничена до 1 раза в 800 мс и только при реальном изменении счёта (throttle)
- Real-time слушатель `addSnapshotListener` читает счёт соперника и передаёт `GameView` через `onGhostSnapshot()`
- По завершении игры статус документа меняется на `finished`, слушатель снимается
- Результат сохраняется в `PlayerStats` (`onlinePvpWins`, `onlinePvpLosses`, `onlinePvpBest`) и в лидерборд (вкладка «PvP»)

**Структура документа матча в Firestore:**
```
matches/{roomId}/
  player1Id:   "uid1"
  player1Name: "Марсель"
  player2Id:   "uid2"
  player2Name: "Tim"
  status:      "waiting" | "playing" | "finished"
  startTime:   epoch_ms
  score1:      0..N
  score2:      0..N
```

## 7. Физика и игровой движок

### 7.1 Рендеринг

Весь игровой рендеринг реализован в `GameView.onDraw(Canvas canvas)` — переопределённом методе `android.view.View`. Порядок слоёв:

1. **Фон** — линейный градиент (`LinearGradient`) — трёхцветный по высоте экрана
2. **Препятствия** — прямоугольники с закруглёнными углами (`drawRoundRect`) в режиме Аркада
3. **Кольцо (задний слой)** — нижняя часть обода, отрисовывается до мяча
4. **Сетка** — цепочка узлов со spring-физикой
5. **Мяч** — PNG-спрайт через `BitmapShader` + `Matrix`
6. **Кольцо (передний слой)** — верхняя часть обода поверх мяча для эффекта глубины
7. **HUD** — счёт, таймер, имя соперника, счёт бота
8. **Эффекты** — комбо-счётчик, вспышка при ударе (`impactFlashUntilMs`), тряска экрана, анимация достижения

**Рисунок 3. Стек слоёв отрисовки `onDraw(Canvas)`:**

```
┌────────────────────────────────────────────────────┐
│  8. Эффекты: комбо / вспышка / тряска / баннер     │  ← поверх всего
├────────────────────────────────────────────────────┤
│  7. HUD: счёт · таймер · имена · счёт бота         │
├────────────────────────────────────────────────────┤
│  6. Кольцо — передний слой (перекрывает мяч)       │
├────────────────────────────────────────────────────┤
│  5. Мяч (BitmapShader + Matrix)                    │
├────────────────────────────────────────────────────┤
│  4. Сетка (spring-узлы, линии между ними)          │
├────────────────────────────────────────────────────┤
│  3. Кольцо — задний слой                           │
├────────────────────────────────────────────────────┤
│  2. Препятствия (Аркада ≥ 3 очков)                 │
├────────────────────────────────────────────────────┤
│  1. Фон (LinearGradient, 3 цвета по высоте)        │  ← основа
└────────────────────────────────────────────────────┘
              ▲ каждый слой рисуется поверх предыдущего
```

Следующий кадр запрашивается через `postInvalidateOnAnimation()` — метод синхронизируется с `Choreographer` и обеспечивает частоту ~60 FPS без занятого ожидания.

### 7.2 Физика броска

Бросок реализован как **квадратичная кривая Безье**:

```java
// P(t) = (1-t)² · P₀ + 2(1-t)t · Pmid + t² · P₁
float inv = 1f - t;
ballX = inv*inv*fromX + 2f*inv*t*midX + t*t*targetX;
ballY = inv*inv*fromY + 2f*inv*t*peakY + t*t*targetY;
```

Параметры вычисляются из жеста свайп:
- `(fromX, fromY)` — начальная точка касания (нижняя часть экрана)
- `(targetX, targetY)` — центр кольца, скорректированный на вектор свайпа
- `midX` — горизонтальная средняя точка дуги
- `peakY` — вершина дуги: `min(fromY, targetY) - velocity * 0.3f`
- `t` — параметр, равномерно растёт от 0 до 1 за время полёта

Такой подход обеспечивает плавную дугу без необходимости интегрировать уравнения движения и гарантирует детерминированный результат при одинаковом свайпе.

**Рисунок 4. Траектория броска — квадратичная кривая Безье:**

```
                        Pmid (вершина дуги)
                           ●
                         /   \
                        /     \
                       /       \
                      /         \
                     /           \
                    /             \
                   /               \
                  /                 \
   t=0 ─────────────────────────────── t=1
       ●                               ●
      P₀ (старт, нижняя часть экрана) P₁ (центр кольца)

    Жест свайп:  ↑ вверх → мяч летит по дуге → кольцо
    peakY = min(fromY, targetY) − velocity × 0.3f
```

### 7.3 Геометрия обода и детекция попадания

Кольцо рендерится как капсула: `rimDrawX`, `rimDrawY`, `rimDrawW`, `rimDrawH` — прямоугольник с закруглёнными торцами. Для коллизий вычисляются дополнительные параметры:

- `rimYTop` — средняя плоскость обода (примерно середина капсулы по высоте)
- `rimYBottom` — нижняя плоскость, через которую мяч должен пройти вниз для засчёта
- `rimInnerLeft` / `rimInnerRight` — внутренние границы проёма (зазор ~78% ширины обода)

Засчитывается попадание при одновременном выполнении условий:
```java
float dx = ballX - (hoopX + hoopWidth / 2f);
boolean inHoop =
    Math.abs(dx) < hoopWidth * 0.48f    // мяч в горизонтальной зоне проёма
    && ballY > rimYTop                  // мяч прошёл уровень обода
    && ballY < rimYBottom               // ещё не выпал из нижней зоны
    && velocityY > 0;                   // мяч летит вниз (нет зачётов снизу вверх)
```

Дополнительно проверяется попадание в стойку кольца (`backboard hit`): если мяч касается боковых граней обода, к скорости по оси X добавляется отскок, визуально имитирующий удар о стойку.

### 7.4 Физика сетки

Сетка реализована как цепочка из `NET_NODES` узлов с **spring-симуляцией**:
- Каждый узел имеет позицию `(x, y)` и скорость `(vx, vy)`
- При попадании мяча — импульс к верхним узлам пропорционально скорости мяча и позиции по дуге
- Горизонтальный импульс зависит от направления вылета: `impulse = 14f * dir + random * 6f`
- Затухание горизонтальной скорости: `vx *= 0.88f` на каждом кадре
- Отрисовка: линии между соседними узлами, цвет зависит от выбранной сетки из инвентаря

### 7.5 Визуальные эффекты

- **Вспышка при попадании** (`impactFlashUntilMs`): полупрозрачный белый оверлей на 70–140 мс; при мощном комбо — длиннее
- **Тряска экрана** (`shakeUntilMs`, `shakeAmpPx`): смещение Canvas через `canvas.translate()` на случайную величину в диапазоне ±ampPx; активируется при комбо ≥5
- **Анимация достижения**: всплывающий баннер на 2500 мс с иконкой мяча/сетки или цветным прямоугольником для фонов
- **Конфетти** (`ConfettiView`): партикловая анимация при победе в дуэли

### 7.6 Вибрация и тактильный отклик

При попадании мяча срабатывает системный вибратор:
- API 26+: `VibrationEffect.createOneShot(50, amplitude)` — стандартизированная амплитуда
- API 21–25: устаревший `vibrator.vibrate(50)` с проверкой версии API во избежание краша

### 8.1 Механика достижений

```java
if (score > 0 && score % 3 == 0 && achievementLevel < 8) {
    achievementLevel++;
    prefs.edit().putInt("achievementLevel", achievementLevel).apply();
    unlockReward(achievementLevel);

```

### 9.1 Интерфейс MatchClient

### 9.2 ИИ-бот (DemoDuelClient)

Бот разработан с целью имитировать поведение реального игрока: он не играет идеально и адаптируется к темпу матча.

**Вероятность промаха (`missChance`):**
```java
float missChance = 0.47f + 0.10f * scoreStreak - 0.05f * missStreak;
if (currentShotType != SHOT_DIRECT) missChance += 0.12f; // банковский бросок сложнее
if (coldStreak > 0) { coldStreak--; missChance = 1.0f; } // «холодная серия»
missChance = clamp(missChance, 0.35f, 0.88f);
```

- Базовый шанс промаха: 47% (~53% попаданий)
- Горячая игра (много голов подряд): шанс промаха растёт до кэпа 88%
- Холодная игра (много промахов подряд): шанс промаха снижается до флора 35%

**Серии (streak system):**
- **Горячая серия:** после 2+ попаданий подряд — 55% вероятность «сломаться» и начать `coldStreak`
- **Холодная серия (`coldStreak`):** бот делает 4–8 обязательных промахов подряд (`missChance = 1.0f`)
- **Разогрев:** после 5+ промахов — 30% вероятность обнуления `scoreStreak` и выхода из спада

**Типы бросков:**

| Тип | Доля | Особенность |
|---|---|---|
| Прямой (`SHOT_DIRECT`) | 72% | Дуга от точки броска прямо к кольцу |
| Банковский левый (`SHOT_BANK_LEFT`) | 14% | Контрольная точка смещается к левой стенке экрана |
| Банковский правый (`SHOT_BANK_RIGHT`) | 14% | Контрольная точка смещается к правой стенке экрана |

**Адаптация темпа (EWMA):**
- Базовый интервал между бросками: 4500 мс
- Интервал адаптируется к темпу игрока через экспоненциальное скользящее среднее: новый_темп = 0.3 × темп_игрока + 0.7 × текущий_темп_бота
- После промаха — дополнительная пауза 600–1300 мс
- Допустимый диапазон: 1800–5500 мс

### 9.3 Реальный онлайн-клиент (OnlinePvpClient)

`OnlinePvpClient` реализует `MatchClient` поверх Firebase Firestore.

**Ключевые механизмы:**
1. **Подписка (`addSnapshotListener`)** — слушает документ матча; при изменении поля `score2`/`score1` соперника вызывает `listener.onGhostSnapshot()`
2. **Отправка счёта (`update`)** — записывает своё поле, но не чаще одного раза в 800 мс и только при реальном изменении счёта:

```java
private static final long SCORE_WRITE_INTERVAL_MS = 800;

@Override
public void sendSnapshot(float x, float y, boolean moving, int score) {
    if (!connected || matchRef == null) return;
    if (score == lastWrittenScore) return;             // дедупликация
    long now = System.currentTimeMillis();
    if (now - lastWriteMs < SCORE_WRITE_INTERVAL_MS) return; // throttle
    lastWrittenScore = score;
    lastWriteMs = now;
    matchRef.update(myScoreField, score);
}
```

3. **Завершение** — обновляет `status: "finished"`, снимает слушатель, освобождает ресурсы

Такое дросселирование существенно снижает количество операций записи в Firestore — критично для контроля расходов на бесплатном тарифе Firebase Spark.

Миграция `Migration(2, 3)` реализована через `ALTER TABLE` — данные существующих пользователей при обновлении приложения не теряются.

### 10.3 Firebase Firestore

Firestore используется в двух независимых контекстах:

**Облачный лидерборд** (`CloudLeaderboardRepository`):
```
leaderboard_scores/{userId}_{mode}/
  mode:       "ARCADE"
  score:      42
  playerName: "Марсель"
  userId:     "abc123"
  timestamp:  1711234567890
```

**Матчмейкинг** (`matchmaking_pool`, `matches`):
```
matchmaking_pool/{userId}/
  userId:     "abc123"
  name:       "Марсель"
  timestamp:  epoch_ms

matches/{roomId}/
  player1Id, player1Name, player2Id, player2Name
  status, startTime, score1, score2
```

`LeaderboardRepository` объединяет локальные (Room) и облачные (Firestore) результаты: дедупликация по имени игрока, сортировка по убыванию очков.

### 10.4 Firebase Auth (Google Sign-In)

`AuthManager` — Singleton, управляющий жизненным циклом аутентификации:

1. `signIn(Activity, RC)` → запускает Google-интент выбора аккаунта
2. `handleSignInResult(data, callback)` → обменивает Google ID Token на Firebase-сессию
3. `isSignedIn()` → проверяет наличие текущего пользователя в `FirebaseAuth.getInstance()`
4. `signOut(activity, callback)` → разрывает сессию Google и Firebase, очищает кэшированный аккаунт
5. При входе автоматически синхронизируется лидерборд с Firestore и активируется возможность матчмейкинга

При недоступности Firebase всё приложение продолжает работать в офлайн-режиме — Google Sign-In является опциональным улучшением, а не обязательным требованием.

### 13.3 Практическая ценность

- APK собирается и запускается на устройствах с Android 5.0–14 без дополнительных зависимостей
- Приложение работает **полностью офлайн** — три режима (Аркада, На время, Дуэль с ботом) не требуют интернета
- При недоступности Firebase матчмейкинг автоматически переключается на ИИ-бота — деградации пользовательского опыта не происходит
- Архитектура с интерфейсом `MatchClient` позволяет заменить Firestore на любой другой backend без изменения `GameView` и игровой логики
- Паттерн Repository изолирует источник данных: одинаковый код Activity работает с Room и Firestore
- Дросселирование записи в Firestore (800 мс) обеспечивает работу в рамках бесплатного тарифа при умеренной нагрузке
- Код опубликован на GitHub с открытым исходным кодом

- Реализации онлайн-матчмейкинга и real-time синхронизации данных через Firestore без WebSocket-сервера и выделенного backend
