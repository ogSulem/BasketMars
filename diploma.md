## Аннотация

Выпускная квалификационная работа посвящена проектированию и разработке мобильного игрового приложения «BasketMars» для платформы Android. В приложении реализованы физически корректная модель броска мяча в кольцо на основе квадратичной кривой Безье, четыре игровых режима (в том числе онлайн-матчмейкинг с реальным соперником через Firebase Firestore), прогрессивная система сложности с динамическими препятствиями, система комбо-бросков, восемь достижений с разблокируемым инвентарем скинов, двуязычный интерфейс и аутентификация через Google Sign-In. Весь игровой движок, физика, анимации и эффекты реализованы средствами нативного Android SDK (Java, Canvas API) без применения сторонних игровых фреймворков. Результатом работы является полнофункциональное приложение, готовое к публикации в Google Play.

Ключевые слова: Android, Java, Canvas API, Firebase Firestore, Room, Google Sign-In, аркадная игра, матчмейкинг, ИИ-бот, квадратичная кривая Безье, паттерны проектирования.

Объем работы: 65 страниц, 10 рисунков, 12 таблиц, 18 источников, 3 приложения.

Во-вторых, разработка игрового движка средствами Android Canvas API представляет собой нетривиальную учебно-исследовательскую задачу, охватывающую большой спектр современных архитектурных практик.

Цель работы – разработать полнофункциональное мобильное приложение игрового жанра «BasketMars» с применением нативного Android SDK, современных архитектурных паттернов, облачных сервисов Firebase и реляционной базы данных Room.

Аркадные игры занимают первое место по числу загрузок (32% от общего числа) [1]. Ключевой тренд последних лет – интеграция соревновательных онлайн-режимов: по данным App Annie, приложения с PvP-режимом демонстрируют на 40% более высокое удержание пользователей на 30-й день по сравнению с одиночными играми [2].

Основные технические вызовы при разработке аркадных игр для Android:

– минимальная задержка рендеринга (целевой FPS – 60 кадров/с);

– честная и предсказуемая физическая модель, обеспечивающая согласованность между визуальным представлением и игровой логикой;

– эффективная работа с сетью без выделенного backend-сервера;

– поддержка широкого диапазона устройств (API 21+, размеры экрана от 5 до 7 дюймов).

### 1.3. Выбор технологического стека

При выборе технологии разработки рассматривались четыре альтернативы (таблица 2).

Таблица 2 – Сравнение технологических стеков для разработки

| Стек | Размер APK | Полный контроль рендеринга | Кривая обучения | Применимость для ВКР |
|---|---|---|---|---|
| Нативный Java/Android SDK | ~6 МБ | Да (Canvas API) | Высокая | Оптимально |
| Unity (C#) | ~50 МБ | Частично | Средняя | Избыточно |
| Flutter (Dart) | ~15 МБ | Ограничен | Средняя | Не оптимально |
| libGDX (Java) | ~12 МБ | Да | Средняя | Излишняя зависимость |

Выбор нативного Java/Android SDK обусловлен тремя причинами.

Во-первых, Canvas API предоставляет прямой доступ к каждому пикселю экрана, что необходимо для реализации физической анимации сетки, эффектов вспышки и тряски экрана.

Во-вторых, нативная разработка обеспечивает минимальный размер APK (около 6 МБ) и наилучшую производительность на диапазоне устройств от low-end до high-end.

В-третьих, исследование низкоуровневых механизмов Android (жизненный цикл Activity, SharedPreferences, Room, Vibrator API, VibrationEffect) является ключевой учебной задачей данной ВКР.

Для облачной части выбрана платформа Firebase (Google): Firebase Firestore обеспечивает real-time синхронизацию без выделенного backend-сервера; Firebase Auth – аутентификацию через Google Sign-In. Локальное хранилище реализовано на базе Room (обертка над SQLite), рекомендованной командой Android Jetpack [3].

## 2. Проектирование приложения BasketMars

### 2.1. Функциональные требования

На основе анализа предметной области сформированы следующие функциональные требования к приложению «BasketMars».

Игровые функции:

– бросок мяча жестом свайп с физически корректной дугой полета;

– детекция попадания в кольцо через геометрию обода (без упрощений);

– четыре игровых режима: Аркада, На время, Дуэль с ботом, Онлайн PvP;

– прогрессивное усложнение в режиме Аркада (препятствия и движущееся кольцо);

– комбо-система с бонусными очками и визуальными эффектами.

Пользовательские функции:

– инвентарь скинов (мячи, сетки, фоны) с разблокировкой за достижения;

– лидерборд (локальный + облачный) с разделением по режимам;

– настройки (имя, музыка, вибрация, язык, аккаунт Google);

– двуязычный интерфейс (русский и английский).

Технические требования:

– минимальная версия Android: API 21 (Android 5.0 Lollipop);

– целевая версия Android: API 34 (Android 14);

– работа без интернета (режимы Аркада, На время, Дуэль);

– аутентификация через Google Sign-In (опционально);

– размер APK: не более 10 МБ.

### 2.3. Проектирование модели данных

Приложение использует два хранилища: локальную базу данных Room и Firebase Firestore. ER-диаграмма локальной базы данных показана на рисунке 4.

```
+-------------------+        +------------------------+
|   player_stats    |        |   leaderboard_entries  |
+-------------------+        +------------------------+
| PK id (LONG = 1)  |        | PK id (LONG, auto)     |
|    totalGames     |        |    mode (String)        |
|    arcadeBest     |        |    score (INT)          |
|    timedBest      |        |    timestamp (LONG)     |
|    duelBest       |        |    playerName (String)  |
|    duelWins       |        +------------------------+
|    duelLosses     |
|    onlinePvpBest  |
|    onlinePvpWins  |
|    onlinePvpLosses|
|    updatedAt      |
+-------------------+
```

*Рисунок 4 – ER-диаграмма локальной базы данных Room*

Сущность PlayerStats (таблица 4) хранится в единственной строке (id = 1) и содержит агрегированную статистику игрока.

Таблица 4 – Поля сущности PlayerStats

| Поле | Тип | Описание |
|---|---|---|
| id | LONG (PK) | Всегда равен 1 (singleton-запись) |
| totalGames | INT | Общее число сыгранных партий |
| arcadeBest | INT | Рекорд в режиме Аркада |
| timedBest | INT | Рекорд в режиме «На время» |
| duelBest | INT | Рекорд в режиме Дуэль |
| duelWins | INT | Победы в Дуэли |
| duelLosses | INT | Поражения в Дуэли |
| onlinePvpBest | INT | Рекорд в Онлайн PvP |
| onlinePvpWins | INT | Победы в Онлайн PvP |
| onlinePvpLosses | INT | Поражения в Онлайн PvP |
| updatedAt | LONG | Временная метка последнего обновления (мс) |

Сущность LeaderboardEntry (таблица 5) хранит каждый результат отдельно; для лидерборда используется агрегирующий запрос MAX(score) GROUP BY playerName.

Таблица 5 – Поля сущности LeaderboardEntry

| Поле | Тип | Описание |
|---|---|---|
| id | LONG (PK, auto) | Автогенерируемый первичный ключ |
| mode | String | Режим игры (ARCADE, TIMED, ONLINE\_DUEL, ONLINE\_PVP) |
| score | INT | Набранные очки |
| timestamp | LONG | Время записи результата (мс с эпохи Unix) |
| playerName | String | Имя игрока (из SharedPreferences) |

Структура коллекции Firebase Firestore для облачного лидерборда:

```
leaderboard_scores/
  {userId}_{mode}/
    mode:        "ARCADE"
    score:       42
    playerName:  "Марсель"
    userId:      "abc123"
    timestamp:   1711234567890
```

Один документ на пару (игрок, режим) обеспечивает эффективные запросы без избыточного хранения дублирующихся записей.

Таблица 6 – Экраны приложения и их назначение

## 3. Реализация приложения

### 3.1. Игровой движок и физика

Ядром приложения является класс GameView, наследующий SurfaceView. Метод onDraw(Canvas) вызывается в цикле через postInvalidate() с целевой частотой 60 Гц. Порядок отрисовки фиксирован: фон – препятствия – кольцо – сетка – мяч – HUD – эффекты.

Физическая модель броска основана на параметрической квадратичной кривой Безье (рисунок 5).

```
                              Pmid (вершина дуги)
                                    *
                                  /   \
                                /       \
                              /           \
                            /               \
                          /                   \
                        /                       \
    t=0 *-----------------------------------------* t=1
       P0 (старт, нижняя часть экрана)   P1 (центр кольца)

    Формула: B(t) = (1-t)^2 * P0 + 2*(1-t)*t * Pmid + t^2 * P1
    Pmid.y = min(P0.y, P1.y) - velocity * 0.3f
```

*Рисунок 5 – Квадратичная кривая Безье для физики броска мяча*

Мяч вычисляется по формуле:

B(t) = (1 – t)² · P0 + 2 · (1 – t) · t · Pmid + t² · P1, где t изменяется от 0 до 1 [5].

После завершения дуги вступает в силу линейная физика под действием гравитации (GRAVITY = 3,5 пикселя/кадр²), что обеспечивает естественный отскок при промахе.

Геометрия обода и детекция попадания. Кольцо отрисовывается как процедурная капсула с параметрами rimDrawX, rimDrawY, rimDrawW, rimDrawH. Для детекции попадания вычисляются дополнительные характеристики (рисунок 6).

```
    +-----------[rim]-----------+    rimYTop (плоскость обода)
    |  rimInnerLeft   rimInnerRight
    |    |                 |
    |    |<----rimW------->|         rimYBottom (нижняя плоскость засчета)
    |    |                 |
```

*Рисунок 6 – Геометрия обода кольца и плоскости детекции попадания*

Попадание засчитывается при одновременном выполнении четырех условий: мяч пересек плоскость rimYTop сверху вниз; центр мяча находится в горизонтальных границах rimInnerLeft..rimInnerRight; мяч прошел через плоскость rimYBottom; вертикальная скорость мяча направлена вниз (velocityY > 0). Такая модель исключает «магические» попадания снизу вверх и обеспечивает соответствие между визуальным представлением кольца и игровой логикой.

Физика сетки реализована как цепочка из NET_NODES узлов со spring-симуляцией: каждый узел имеет позицию (x, y) и скорость (vx, vy); при попадании мяча верхним узлам передается импульс пропорционально скорости и позиции; горизонтальная скорость затухает с коэффициентом 0,88 за кадр.

Вибрация при попадании реализована с учетом версии API: на API 26+ используется VibrationEffect.createOneShot(50, amplitude) с нормированной амплитудой; на API 21–25 применяется устаревший метод vibrate(50).

### 3.3. ИИ-бот (DemoDuelClient)

Класс DemoDuelClient реализует интерфейс MatchClient и запускается в главном потоке через Handler/Looper для совместимости с GameView. Бот разработан с целью имитировать поведение реального игрока: отсутствие идеальной точности, серии «горячей» и «холодной» игры, адаптация темпа.

Вероятность промаха вычисляется по формуле:

missChance = 0,47 + 0,10 · scoreStreak – 0,05 · missStreak + 0,12 · [банковский бросок]

с ограничением [0,35; 0,88]. При активной «холодной серии» (coldStreak > 0) вероятность промаха принудительно устанавливается равной 1,0 (промах гарантирован).

Бот поддерживает три типа бросков (таблица 8).

Таблица 8 – Типы бросков ИИ-бота и их параметры

| Тип броска | Доля, % | Описание | Дополнительный шанс промаха |
|---|---|---|---|
| Прямой (SHOT_DIRECT) | 72 | Дуга от позиции броска прямо к кольцу | 0% |
| Банковский левый (SHOT_BANK_LEFT) | 14 | Контрольная точка смещена к левому краю | +12% |
| Банковский правый (SHOT_BANK_RIGHT) | 14 | Контрольная точка смещена к правому краю | +12% |

### 3.4. Онлайн-мультиплеер через Firebase

Онлайн-матчмейкинг реализован через Firebase Firestore без выделенного backend-сервера. Последовательность установки соединения показана на рисунке 9.

```
  Игрок A         MatchmakingActivity        Firestore          Игрок B
     |                    |                     |                   |
     |-- выбрать PvP ---->|                     |                   |
     |                    |-- PUT pool/{myId} ->|                   |
     |                    |   {searching}       |<-- PUT pool/{B} --|
     |                    |<-- snapshot: B -----+                   |
     |                    |-- CREATE match/ --->|                   |
     |                    |   {p1,p2,status}    |--- snapshot ----->|
     |                    |-- DELETE pool ----->|<--- DELETE pool --|
     |<--- GameActivity --|                     |--- GameActivity ->|
     |   ONLINE_PVP       |                     |   ONLINE_PVP      |
     |-- sendSnapshot()------------------------------------------>  |
     |<-- onGhostSnapshot() --------------------------------------- |
```

*Рисунок 9 – Диаграмма последовательности онлайн-матчмейкинга*

Алгоритм матчмейкинга:

– игрок A записывает себя в коллекцию matchmaking_pool со статусом «searching»;

– подписка на коллекцию обнаруживает запись игрока B;

– первый нашедший соперника создает документ матча в коллекции matches;

– оба игрока переходят в GameActivity с параметрами roomId и myRole;

– если за 30 секунд соперник не найден – предлагается сыграть против бота.

Класс OnlinePvpClient реализует интерфейс MatchClient: метод sendSnapshot() записывает позицию и счет в Firestore с ограничением частоты 100 мс; Listener-подписка доставляет обновления соперника в основной поток через addSnapshotListener.

// GameView.java — честная детекция попадания в кольцо
private void handleFairScoring(float prevX, float prevY) {
    if (gameOver || scored || velocityY <= 0) return;

    float left  = rimInnerLeft  + ballRadius * 0.25f;
    float right = rimInnerRight - ballRadius * 0.25f;
    if (left >= right) return;

    float prevBottom = prevY + ballRadius;
    float currBottom = ballY + ballRadius;
    boolean crossesTopDown = prevBottom < rimYTop && currBottom >= rimYTop;
    if (crossesTopDown && ballX > left && ballX < right) {
        enteredHoop = true;
    }

    if (enteredHoop) {
        float prevTop = prevY - ballRadius;
        float currTop = ballY - ballRadius;
        boolean passesThrough = prevTop < rimYBottom && currTop >= rimYBottom;
        if (passesThrough && ballX > left && ballX < right) {
            onScored();
        }
    }
}

// GameView.java — вычисление геометрии обода кольца
private void computeHoopGeometry() {
    rimDrawH      = hoopHeight * 0.10f;
    rimDrawW      = hoopWidth  * 0.92f;
    rimDrawX      = hoopX + (hoopWidth - rimDrawW) / 2f;
    rimDrawY      = hoopY + hoopHeight * 0.14f;
    rimYTop       = rimDrawY + rimDrawH * 0.55f;
    rimYBottom    = rimDrawY + rimDrawH * 2.15f;
    rimInnerLeft  = rimDrawX + rimDrawW * 0.11f;
    rimInnerRight = rimDrawX + rimDrawW * 0.89f;
    rimRadius     = Math.max(8f, Math.min(22f, rimDrawH * 0.18f));
}

    playerName TEXT    NOT NULL

-- Индекс для быстрого получения топ-N по режиму

-- Запрос топ-10 по режиму (агрегация лучшего результата на игрока)
SELECT playerName, mode, MAX(score) AS score,
       MIN(timestamp) AS timestamp, 0 AS id
FROM leaderboard_entries
WHERE mode = ?
GROUP BY playerName, mode
ORDER BY score DESC, timestamp ASC
