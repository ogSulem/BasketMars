## Аннотация

Выпускная квалификационная работа посвящена проектированию и разработке мобильного игрового приложения «BasketMars» для платформы Android. В приложении реализованы физически корректная модель броска мяча в кольцо на основе квадратичной кривой Безье, четыре игровых режима (в том числе онлайн-матчмейкинг с реальным соперником через Firebase Firestore), прогрессивная система сложности с динамическими препятствиями, система комбо-бросков, восемь достижений с разблокируемым инвентарём скинов, двуязычный интерфейс (русский и английский) и аутентификация через Google Sign-In. Весь игровой движок, физика, анимации и эффекты реализованы средствами нативного Android SDK (Java, Canvas API) без применения сторонних игровых фреймворков. Хранение данных осуществляется посредством локальной реляционной базы данных Room (SQLite) и облачного хранилища Firebase Firestore. Результатом работы является полнофункциональное приложение, готовое к публикации в Google Play.

Ключевые слова: Android, Java, Canvas API, Firebase Firestore, Room, Google Sign-In, аркадная игра, матчмейкинг, ИИ-бот, квадратичная кривая Безье, паттерны проектирования.

Объём работы: 72 страницы, 15 рисунков, 12 таблиц, 18 источников, 3 приложения.

## Введение

Мобильный рынок игровых приложений является одним из наиболее быстро растущих сегментов IT-индустрии. По данным аналитической компании Sensor Tower, совокупная выручка от мобильных игр в 2023 году превысила 100 млрд долларов США [18]. Жанр аркадных игр занимает лидирующую позицию по количеству установок: механика «ещё одного броска» обеспечивает высокий удерживающий потенциал при минимальном пороге вхождения [17].

Актуальность данной работы определяется двумя факторами. Во-первых, среди существующих мобильных баскетбольных игр практически отсутствуют нативные Android-решения с честной физической детекцией попадания, онлайн-матчмейкингом без выделенного backend-сервера и полноценной офлайн-работой. Во-вторых, разработка игрового движка средствами Android Canvas API представляет собой нетривиальную учебно-исследовательскую задачу, охватывающую широкий спектр современных архитектурных практик.

Цель работы – разработать полнофункциональное мобильное приложение игрового жанра «BasketMars» с применением нативного Android SDK, современных архитектурных паттернов, облачных сервисов Firebase и реляционной базы данных Room.

Для достижения цели поставлены следующие задачи:

1. Провести анализ рынка мобильных игр и существующих аналогов.
2. Обосновать выбор технологического стека.
3. Спроектировать многослойную архитектуру приложения.
4. Реализовать игровой движок на базе Canvas API без сторонних фреймворков.
5. Разработать физически корректную модель броска на основе квадратичной кривой Безье.
6. Создать четыре игровых режима, в том числе онлайн-матчмейкинг через Firebase Firestore.
7. Реализовать ИИ-бота с человекоподобным поведением.
8. Разработать систему достижений и разблокируемых скинов.
9. Интегрировать локальное хранилище Room и облачный лидерборд.
10. Реализовать аутентификацию пользователя через Google Sign-In и управление профилем.

13. Провести тестирование и оценку качества приложения.

Объект исследования – процесс разработки нативного мобильного игрового приложения для платформы Android.

Аркадные игры занимают первое место по числу загрузок (32% от общего числа) [18]. Ключевой тренд последних лет – интеграция соревновательных онлайн-режимов: по данным App Annie, приложения с PvP-режимом демонстрируют на 40% более высокое удержание пользователей на 30-й день по сравнению с одиночными играми [17].

Основные технические вызовы при разработке аркадных игр для Android:

– минимальная задержка рендеринга (целевой показатель – 60 кадров/с);

– честная и предсказуемая физическая модель, обеспечивающая согласованность между визуальным представлением и игровой логикой;

– эффективная работа с сетью без выделенного backend-сервера;

– поддержка широкого диапазона устройств (API 21+, размеры экрана от 5 до 7 дюймов).

Рисунок 2 – Архитектурно-технологическая диаграмма приложения BasketMars

Ключевые архитектурные решения:

– **Паттерн Repository**: класс `LeaderboardRepository` абстрагирует источники данных (Room и Firestore) от слоя UI. Приложение взаимодействует только с репозиторием, не зная о конкретной реализации хранилища.

– **Паттерн Strategy через интерфейс `MatchClient`**: алгоритм поведения соперника подменяется без изменения кода `GameView`. `DemoDuelClient` реализует ИИ-бота, `OnlinePvpClient` – реального соперника через Firestore. Оба реализуют единый интерфейс `MatchClient`.

– **Паттерн Singleton**: `AuthManager` и `MusicPlayer` существуют в единственном экземпляре на протяжении жизни приложения.

– **Application-уровень инициализации**: класс `BasketballGameApp` (наследник `Application`) инициализирует Firebase, Room и репозиторий один раз при старте процесса.

## 3. Реализация приложения

### 3.1. Игровой движок и физика

Игровой движок реализован в классе `GameView`, наследующем стандартный `android.view.View`. Такой подход позволяет встраивать игровое поле в любой `ViewGroup` без дополнительных фреймворков. Игровой цикл основан на механизме `invalidate() → onDraw()`, вызываемом каждые 16 мс (целевые 60 кадров/с).

**Модель броска на основе квадратичной кривой Безье.** Траектория мяча описывается кривой Безье второго порядка:

```
B(t) = (1-t)² · P0 + 2t(1-t) · P1 + t² · P2,   t ∈ [0, 1]
```

где P0 – начальная позиция мяча, P2 – конечная точка (рассчитывается исходя из направления и силы свайпа), P1 – управляющая точка (формирует характерный «дугообразный» изгиб). Параметр t дискретизируется с шагом 0.02, что даёт 50 точек траектории – достаточно для плавной анимации.

Преимущество перед классической параболой: управляющая точка P1 позволяет тонко настраивать «форму» броска в зависимости от игрового режима и уровня сложности, сохраняя визуальную предсказуемость для игрока.

**Детекция попадания («честная физика»).** Попадание определяется не по конечной точке кривой, а по геометрии обода кольца: мяч считается попавшим, если его траектория проходит сквозь внутреннее отверстие обода при условии отсутствия столкновения с металлическими краями. Данный подход устраняет «несправедливые» попадания, когда мяч визуально проходит мимо кольца, но засчитывается системой.

**Система комбо-бросков.** Каждое последовательное попадание инкрементирует счётчик `comboStreak`. При значении 3 и выше на экране отображается анимированный индикатор комбо с эффектом свечения. Значение `bestComboStreak` фиксирует рекорд сессии.

**Визуальные эффекты.** В игровом движке реализованы следующие эффекты: тряска экрана при касании обода (`shakeUntilMs`, `shakeAmpPx`); тряска кольца при касании (`rimShakeUntilMs`); вспышка при попадании (`impactFlashUntilMs`); волновая анимация сетки кольца (7 узлов, `netDx/netVx/netDy/netVy`); эффект конфетти (`ConfettiView`, 90 частиц, 10 цветов, 4 секунды). Вибрация реализована с учётом версии API:

```java
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.S) {
    VibratorManager vm = (VibratorManager)
        ctx.getSystemService(Context.VIBRATOR_MANAGER_SERVICE);
    if (vm != null) vm.getDefaultVibrator()
        .vibrate(VibrationEffect.createOneShot(40,
            VibrationEffect.DEFAULT_AMPLITUDE));
} else if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O) {
    Vibrator v = (Vibrator) ctx.getSystemService(Context.VIBRATOR_SERVICE);
    if (v != null) v.vibrate(
        VibrationEffect.createOneShot(40,
            VibrationEffect.DEFAULT_AMPLITUDE));
}
```

### 3.2. Реализация игровых режимов

В приложении реализованы четыре игровых режима, описанных перечислением `GameMode`:

– **ARCADE** – бесконечный режим с прогрессивным усложнением. По мере роста счёта появляются горизонтально движущиеся препятствия (`showObstacle`, `showObstacle2`) и начинает двигаться кольцо (`movingHoop`, `hoopVX`);

– **TIMED** – ограничение по времени 60 секунд (`modeDurationMs = 60_000`). Отображается обратный таймер; по истечении времени вызывается `sessionListener.onSessionComplete()`;

– **ONLINE_DUEL** – дуэль против ИИ-бота `DemoDuelClient` с синхронизацией через интерфейс `MatchClient`;

– **ONLINE_PVP** – дуэль против реального игрока через Firebase Firestore, с матчмейкингом в `MatchmakingActivity`.

Таблица 6 – Сравнение игровых режимов

| Параметр | ARCADE | TIMED | ONLINE_DUEL | ONLINE_PVP |
|---|---|---|---|---|
| Длительность | Бесконечно | 60 с | 60 с | 60 с |
| Соперник | Нет | Нет | ИИ-бот | Реальный игрок |
| Препятствия | Да (динамические) | Нет | Нет | Нет |
| Движущееся кольцо | Да | Нет | Нет | Нет |
| Сохранение в Room | Да | Да | Да | Да |
| Сохранение в Firestore | Да (если авторизован) | Да | Да | Да |

После завершения игры `GameActivity.handleSessionComplete()` вызывает `LeaderboardRepository.saveScore()` и `updateStats()`, сохраняя результаты в Room и Firestore. Статистика (`PlayerStats`) обновляется в фоновом IO-потоке через `ExecutorService`.

### 3.3. ИИ-бот (DemoDuelClient)

`DemoDuelClient` реализует интерфейс `MatchClient` и имитирует поведение реального игрока-соперника. Ключевые особенности реализации:

**Адаптация темпа.** Бот отслеживает скорость набора очков реальным игроком через метрику `playerScorePerSec`, вычисляемую с помощью скользящего среднего (EWMA):

```java
playerScorePerSec = 0.75f * playerScorePerSec + 0.25f * instRate;
```

Коэффициент сглаживания 0.75 обеспечивает устойчивость к резким скачкам темпа.

**Система «холодных серий».** После 2+ попаданий подряд с вероятностью 55% активируется «холодная серия» из 4–8 обязательных промахов (`coldStreak`), что имитирует психологические «спады» реального игрока.

**Банковские броски.** С вероятностью 28% бот выбирает траекторию «от стенки» (банковский бросок): сначала движется к левой или правой границе экрана, затем к кольцу. Это создаёт разнообразие визуального поведения призрака.

**Параметры вероятностей:**

– базовый шанс промаха: 47%;

– максимальный шанс промаха: 88%;

– минимальный шанс промаха: 35%;

– дополнительный штраф за банковский бросок: +12%;

– интервал между бросками: 1800–5500 мс.

Позиция призрака бота обновляется каждые 16 мс через `Handler.postDelayed()` и передаётся в `GameView` через `applyGhostSnapshot()`.

### 3.4. Онлайн-мультиплеер через Firebase

**Матчмейкинг.** Алгоритм поиска соперника в `MatchmakingActivity` работает следующим образом (рисунок 6):

```
Игрок А                  Firestore                    Игрок Б
   │                         │                            │
   │─ set(pool/A, searching)─▶│                            │
   │                         │◀─ set(pool/B, searching) ──│
   │                         │                            │
   │◀─ snapshot(pool/*)  ────│                            │
   │                         │                            │
   │─ transaction: ──────────▶│                            │
   │   create match/{roomId}  │                            │
   │   update pool/A matched  │                            │
   │   update pool/B matched  │                            │
   │                         │                            │
   │                         │─── snapshot(pool/B) ──────▶│
   │                         │    status=matched          │
   │  enterGame(role=1) ─────│                            │
   │                         │               enterGame(role=2)
```

Рисунок 6 – Схема алгоритма матчмейкинга

Транзакция Firestore гарантирует атомарность: два игрока не могут одновременно создать два разных матча. Если за 30 секунд соперник не найден, происходит автоматический откат к режиму ONLINE_DUEL (бот).

**Синхронизация счёта.** `OnlinePvpClient` подписывается на документ матча через `addSnapshotListener`. При каждом обновлении счёта соперника в Firestore колбэк `onGhostSnapshot()` вызывается на главном потоке. Для предотвращения избыточных записей в Firestore введено ограничение частоты (`SCORE_WRITE_INTERVAL_MS = 800 мс`).

### 3.6. Локализация и настройки

Двуязычная поддержка реализована через стандартный механизм Android Resource Qualifiers: ресурсы на русском языке находятся в `res/values/strings.xml`, на английском – в `res/values-en/strings.xml`. Переключение языка производится в `SettingsActivity` с сохранением выбора в `SharedPreferences`.

Класс `LocaleHelper` применяет сохранённую локаль через метод `attachBaseContext`:

```java
public static Context wrap(Context ctx) {
    String lang = getSavedLang(ctx);
    Locale locale = new Locale(lang);
    Locale.setDefault(locale);
    Configuration config = new Configuration(
        ctx.getResources().getConfiguration());
    config.setLocale(locale);
    return ctx.createConfigurationContext(config);
}
```

Метод `attachBaseContext()` переопределён во всех семи Activity приложения.

Музыкальный плеер (`MusicPlayer`) работает с медиафайлами из `res/raw` (обнаружение происходит через Reflection API класса `R$raw`). Поддерживается перелистывание треков, пауза и возобновление воспроизведения при сворачивании/разворачивании приложения через `ActivityLifecycleCallbacks`.

Настройки аккаунта Google Sign-In реализованы в `SettingsActivity`. При успешной аутентификации Firebase UID сохраняется в `AuthManager` (singleton) и используется для атрибуции записей в облачном лидерборде. Не авторизованные пользователи получают анонимный UUID (`anonymousId`), сохранённый в `SharedPreferences`.

### Реализация паттерна Repository и объединения источников данных

Фрагмент класса `LeaderboardRepository`, демонстрирующий объединение облачных и локальных данных:

```java
public void getTopScores(String mode, int limit,
        ListCallback<List<LeaderboardEntry>> callback) {
    if (cloudRepo != null && cloudRepo.isAvailable()) {
        cloudRepo.getTopScores(mode, limit, cloudEntries ->
            ioExecutor.execute(() -> {
                List<LeaderboardEntry> local =
                    leaderboardDao.getTop(mode, limit);
                List<LeaderboardEntry> merged =
                    merge(local, cloudEntries, limit);
                if (callback != null) callback.onResult(merged);
            }));
    } else {
        ioExecutor.execute(() -> {
            List<LeaderboardEntry> entries =
                leaderboardDao.getTop(mode, limit);
            if (callback != null) callback.onResult(entries);
        });
    }
}

private List<LeaderboardEntry> merge(
        List<LeaderboardEntry> local,
        List<LeaderboardEntry> cloud, int limit) {
    Map<String, LeaderboardEntry> best = new LinkedHashMap<>();
    for (LeaderboardEntry e : cloud)
        if (e.playerName != null) best.put(e.playerName, e);
    for (LeaderboardEntry e : local) {
        if (e.playerName == null) continue;
        LeaderboardEntry ex = best.get(e.playerName);
        if (ex == null || e.score > ex.score)
            best.put(e.playerName, e);
    }
    List<LeaderboardEntry> result = new ArrayList<>(best.values());
    result.sort((a, b) -> Integer.compare(b.score, a.score));
    return result.subList(0, Math.min(result.size(), limit));
}
```
